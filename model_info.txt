2) üìå Key Concept: What is ASSI?

ASSI measures total Aadhaar service workload at a location.

ASSI formula:
ASSI = 0.5‚ãÖEnrolment + 0.3‚ãÖDemographicUpdates + 0.2‚ãÖBiometricUpdates

Interpretation:

- Enrolment is highest effort (full Aadhaar creation process)
- Demographic updates are medium effort
- Biometric updates are smaller but important periodic workload

So forecasting ASSI directly predicts service center stress/load.

6) üß† Models Used (What approach we follow)

We use a 2-model setup:


Model A ‚Äî Baseline (for benchmark)

‚úÖ Simple Moving Average (SMA-3)

yhat(t+1) = mean( y(t), y(t‚àí1), y(t‚àí2) )

Purpose:
- provides a baseline benchmark
- used to show improvement


Model B ‚Äî Main Model (high impact)

‚úÖ Prophet Forecasting Model (recommended)

Prophet learns:
- trend
- seasonal patterns
- local changes (changepoints)

Prophet output includes:
- prediction (yhat)
- confidence interval (yhat_lower, yhat_upper)


7) üìå Training workflow (How to implement)

Step 1 ‚Äî Load data

- Read master_monthly.csv
- Convert month_year to date:

ds = month_year + '-01' converted to datetime

Prophet requires:
- ds = date column
- y = target column (ASSI)


Step 2 ‚Äî Select hotspots

- Select top 20 from:
./outputs/tables/top25_mean_assi.csv

If not available:
- Compute mean ASSI by (state, district, pincode) and select top 20.


Step 3 ‚Äî Backtesting (mandatory)

To show model quality, evaluate using time-based split:

Train set: all months except last 2 months
Test set: last 2 months

Train Prophet on train set.
Predict test set.
Compute error.


8) ‚úÖ Evaluation metrics

Use forecasting metrics:

SMAPE (preferred)

SMAPE = (100% / n) * Œ£ [ |y - yhat| / ( (|y| + |yhat|)/2 ) ]

Why:
- stable even when values are small

Optionally:
- MAPE (with +1 stabilization)

We will report:
- Prophet SMAPE
- baseline SMA SMAPE
